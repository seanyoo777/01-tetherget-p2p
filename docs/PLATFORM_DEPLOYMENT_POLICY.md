# 플랫폼 배포·점검·Feature Flag 정책 (공통)

**범위**: TetherGet 제품군 및 연계 허브(GameHub, StreamHub, OneAI 등)를 **공통 운영 관점**에서 정리한 **정책 문서**입니다.  
**금지**: 본 문서는 **CI/CD 파이프라인 구현**, **실서버 설정 변경**, **DB 마이그레이션 실행**, **외부 API 직접 연동**을 포함하지 않습니다. 실제 적용은 각 저장소·인프라 담당과 별도 절차로 진행합니다.

**저장소**: `01-TetherGet-P2P`에 두되, 모노레포·다른 번호 저장소와 **원칙 공유**용으로 사용할 수 있습니다.

---

## 1. Branch 전략

| 브랜치 | 용도 | 병합·보호 원칙 |
|--------|------|----------------|
| **`main`** | 프로덕션과 동일 또는 직전 릴리스 후보를 반영하는 **안정 트렁크** | 직접 푸시 금지(권장). PR 필수, 리뷰·CI 통과 후만 병합. 태그·릴리스 노트와 연동. |
| **`dev`** | 일상 통합·QA 시작점 | `feature/*` 병합 허브. nightly/통합 빌드 기준. |
| **`staging`** | 프로덕션과 동일한 구성에 가깝게 띄운 **사전 검증** 환경에 매핑되는 브랜치(또는 태그) | `dev`에서 승격. 프로덕션 배포 직전 **스모크·회귀** 전용. |
| **`feature/*`** | 단일 이슈·단일 기능 단위 | 수명 짧게 유지. 완료 후 `dev`로 PR, 장기 분기 지양. |

**추가 권장**

- **`hotfix/*`**: 프로덕션 긴급 수정만. `main`에서 분기 → 수정 → `main` + `dev`(및 필요 시 `staging`)에 역병합.
- **릴리스 브랜치 `release/x.y`**: 버전 동결·문서·changelog만 모을 때 선택적으로 사용.

## 2. 배포 정책

### 2.1 Preview 배포

- **목적**: PR 단위 또는 `dev` 스냅샷을 **격리 URL**에서 검증.
- **원칙**: 프로덕션 DB·프로덕션 시크릿과 **분리**. 목업 또는 읽기 전용 스냅샷 권장.
- **트리거**: PR 오픈/업데이트, 또는 수동 워크플로 실행(구현은 각 팀 인프라에 위임).

### 2.2 Production 승인

- **게이트**: `staging`에서 **체크리스트**(스모크, 크리티컬 경로, 관리자·결제 인접 기능은 별도 항목) 통과 후에만 프로덕션 배포 승인.
- **역할**: 최소 1인 **코드 리뷰** + 운영/기획 중 **릴리스 승인자** 1명(조직에 맞게 정의).
- **문서**: 배포 시각, 버전(커밋 SHA/태그), 변경 요약, 롤백 포인트를 **릴리스 노트** 또는 운영 로그에 기록.

### 2.3 Rollback

- **1순위**: 이전에 검증된 **아티팩트(이미지/번들) 재배포** 또는 **트래픽을 이전 버전으로 스위치**(블루그린·카나리 정책이 있을 경우).
- **2순위**: **Feature Flag**로 신기능만 즉시 OFF(코드 롤백보다 빠른 경우).
- **금지된 가정**: DB 스키마가 하위 호환되지 않은 상태에서 “앱만” 롤백하는 것은 **데이터 불일치**를 유발할 수 있으므로, DB 정책(§5)과 함께 설계할 것.

## 3. Feature Flag 정책

### 3.1 기능 숨김

- **용도**: 미완성 UI, 실험 기능, 규제·리스크 검토 전 기능.
- **원칙**: 기본값은 **OFF** 또는 “내부 역할만 ON”. 라우트·메뉴·API 응답에서 **일관되게** 숨김(프론트만 가리면 안 되는 경우 서버에서도 거부).

### 3.2 관리자 ON/OFF

- **용도**: 운영 중 긴급 차단, 점검 모드, 특정 테넌트/채널만 개방.
- **원칙**: 플래그 변경 시 **감사 로그**(누가, 언제, 이전/이후 값). 가능하면 **관리자 화면 + 서버 소스 오브 트루스** 이중이 아닌 **단일 소스**를 권장.

### 3.3 점진 배포

- **용도**: 카나리(일부 사용자·일부 리전·일부 인스턴스만 신버전).
- **원칙**: 플래그와 트래픽 분할 정책을 **문서화**; 지표(에러율, 지연, 비즈니스 KPI) **임계치 초과 시 자동 되돌림** 여부를 사전에 정의.

---

## 4. 점검 정책

### 4.1 새벽 점검

- **권장**: 트래픽·거래 **저부하 시간대**에 인프라·DB·대규모 배포를 배치.
- **절차**: 사전 **N일 공지**, 점검 창 **시작·종료 시각(UTC+한국 기준 명시)** , 영향 범위(읽기 전용/쓰기 중단 등) 명시.

### 4.2 거래·게임 중 점검 제한

- **P2P·정산·에스크로**: 진행 중 주문이 있는 경우 **쓰기 중단형 점검**은 최소화. 불가피 시 **주문 상태 머신**과 호환되는 순서(§5 migration 순서와 연계)로만 수행.
- **GameHub / 실시간 세션**: 매치·라운드 진행 중 **강제 종료** 지양. 필요 시 “신규 매치만 중단”, “기존 세션은 드레인” 정책을 우선 검토.

### 4.3 공지 배너

- **트리거**: 예정 점검, 지연, 사고, 부분 장애.
- **원칙**: **시작 전** 배너 노출, 점검 종료 후 **명시적 해제**; 모바일/PWA(§9)에서도 동일 메시지 소스를 사용할 것을 권장.

## 5. DB 변경 정책

### 5.1 Backward compatibility

- **Expand → Migrate → Contract** 패턴 권장:  
  1) 신규 컬럼·테이블 **추가**(구버전 앱이 깨지지 않음)  
  2) 데이터 **이전·듀얼 쓰기**  
  3) 구 스키마·코드 경로 **제거**
- **호환 깨는 변경**(NOT NULL 추가, 타입 축소 등)은 **다단계 릴리스** 또는 **짧은 점검 창**과 함께 계획.

### 5.2 Migration 순서

1. **배포 전** 가능한 변경: additive DDL, 인덱스(온라인 인덱스 가능 시), 기본값이 안전한 신규 컬럼.
2. **앱 배포**: 신 코드가 신·구 스키마 모두 읽을 수 있게.
3. **데이터 백필** 완료 확인 후, 구 컬럼 의존 제거 및 **contract** 마이그레이션.
4. **롤백**: 롤백 시 앱·DB 중 **어느 쪽을 기준**으로 되돌릴지 **릴리스별로** 문서에 적어 둠.

## 6. 실시간 서비스 정책 (플랫폼별)

공통: **연결 끊김·재연결·아이들 타임아웃**은 제품별로 수치는 다르되, “배포 중 드레인 vs 강제 킥”을 사전에 정의한다.

| 플랫폼 | 배포 시 고려 | 비고 |
|--------|----------------|------|
| **GameHub** | 매치메이킹·게임 서버 인스턴스 **롤링 재시작** 권장; 진행 중 세션 보호 정책(드레인 시간) | 버전 불일치 클라이언트 처리(강제 업데이트 vs 호환) 명시 |
| **StreamHub** | 푸시·구독 **sticky** 여부, 재연결 스톰 방지 | 대량 이벤트 시 백프레셔·큐 적체 모니터링 |
| **OneAI** | 모델·프롬프트 버전 **플래그**로 점진 전환; 타임아웃·폴백 문구 | 비용·레이트 리밋 급변 시 플래그로 모델 스위치 |
| **TetherGet (P2P)** | 주문·에스크로·지갑 인접 경로는 **플래그 + 호환 API** 우선; 관리자 기능은 §3.2와 연동 | 본 저장소(01) 기준 mock-first·보안 우선 원칙은 [AGENTS.md](../AGENTS.md) 참고 |

## 7. 긴급 패치 정책

- **트리거**: 보안 취약점, 자금·주문 오류, 개인정보 유출 위험, 가용성 급락.
- **절차**: `hotfix/*` → 최소 변경 → `main` 반영 → **사후** `dev` 등에 역병합; 사후 리뷰(24~48시간 내) 권장.
- **소통**: 내부 인시던트 채널 + 사용자 공지(심각도에 따라 §4.3).
- **데이터**: 긴급이라도 **가능한 한** 마이그레이션은 additive만; 불가피한 contract 변경은 롤백 시나리오를 **반드시** 1페이지로 적어 승인.

---

## 8. 로그·모니터링 정책

- **구조화 로그**: 요청 ID, 사용자·세션(PII 최소화), 버전, 플래그 스냅샷(필요 시).
- **알림**: SLO 위반, 에러율 급증, 큐 적체, DB 커넥션 고갈 등 **단계별**(경고 → 장애 → 전사).
- **보존 기간**: 규제·비용에 맞게 보존·마스킹 정책을 별도 보안 문서와 정합.
- **배포 상관**: 배포 직후 **canary 구간**은 샘플링 레이트를 유지하거나 일시 상향해 회귀를 빠르게 포착.

---

## 9. 모바일·PWA 업데이트 정책

- **캐시**: Service Worker·정적 자산 버저닝; **강제 리프레시**가 필요한 보안 패치는 공지 + `Cache-Control` 정책을 문서화.
- **호환**: 최소 지원 OS/브라우저 버전 상향 시 **사전 공지 기간** 명시.
- **스토어**: 네이티브 앱이 있는 경우 스토어 심사 지연을 고려해 **플래그로 서버 기능**을 먼저 내리는 등 클라보다 서버가 앞서지 않도록 조율.
